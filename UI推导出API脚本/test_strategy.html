测试策略

UI 自动化 + API 自动化，为什么缺一不可

测试架构组 · 2026-06
01 · 仅有 UI 自动化时
三个固有短板
SLOW · 速度慢

每个场景启动浏览器、等待渲染，全量回归需数小时；发布窗口紧时无法快速验收。

NOISY · 误报多

UI 元素位置变动、加载时序抖动均可导致用例失败，但业务逻辑完全正常。误报让人对报告失去信任。

COUPLED · 数据准备慢

前置数据只能通过页面操作创建，步骤多、耗时长，前置失败会污染目标场景的结论。

BLIND · 存在盲区

前端有校验不代表接口有校验。绕过前端直接调接口可触达的风险，UI 用例完全覆盖不到。

02 · API 自动化补位后
对应解法
FAST · 毫秒级反馈

同一场景 API 执行速度比 UI 快 10–20 倍，全量回归从小时缩短到分钟级。

SIGNAL · 失败即真实问题

API 用例不感知 UI 变动，挂了就是业务逻辑或接口本身出了问题，定位层次清晰。

CLEAN · 接口级数据准备

Given 阶段直接调接口创建前置数据，百毫秒完成，独立干净，不依赖页面状态。

DEEP · 覆盖后端真实逻辑

验证接口层本身的业务约束，UI 层有无校验不影响结论，后端安全边界有据可查。

03 · 自动派生链路
API 用例从 UI 用例自动生成，
追加成本≈ Review
结论： UI 告诉你"有没有问题"，API 告诉你"问题在哪层"。双层覆盖，排查成本减半，而 API 用例由链路自动派生，额外人力投入仅为 Review。
